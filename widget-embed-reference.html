Pilot integration guide

Widget Embed Reference — One Script Tag, Any Page

Everything an engineer needs to put the Veru365 chat widget on a client's website and on any number of separate landing pages: the tag, its attributes, what it renders, and exactly which parts are production-hardened versus pilot-scoped today.

Quick start

The one-line install

The entire integration is a single <script> tag. Paste it before the closing </body> of any page — a homepage, a campaign landing page, a template used by a thousand pages — and a branded chat bubble appears bottom-right. There is no per-page registration, no SDK, no build step.

Copy-paste block (live demo environment)

This exact tag runs on the demo hub, the three Acme Wealth pages, and this very page — view source and check:

<script src="https://ihf-frontend-staging-mr64uhwn4a-uc.a.run.app/widget.js"
        data-org="veru365"
        data-base="https://ihf-frontend-staging-mr64uhwn4a-uc.a.run.app" defer></script>

The Summit Planning demo page carries the identical tag with data-org="summit-planning" — that one attribute is the entire difference between the two brands. (The data-base attribute here is redundant — it equals the script's own origin, which is the default — the demo pages just spell it out.) The general form, as documented in the script itself:

<script src="https://app.example.com/widget.js" data-org="acme" defer></script>
  • Works identically on N pages. The same tag goes on the homepage and on every separate landing page — each page gets its own independent chat session.
  • defer is recommended so the script never blocks page rendering. The chat iframe itself is lazy-loaded only when a visitor first clicks the bubble, so the widget costs the host page essentially nothing until it's used.
  • Safe to template blindly. A second identical tag on the same page is a no-op (double-injection guard), and every network call inside the widget is wrapped so a failure can never break the host page.
Attribute reference

Every attribute the script reads

Two attributes total. One selects the client brand, the other selects the environment.

AttributeRequiredType / formatDefaultEffect
data-org Yes Org slug — lowercase letters, digits, hyphens; must start and end alphanumeric; max 64 chars (validated server-side as ^[a-z0-9][a-z0-9-]{0,63}$) none — if missing, the widget logs a console warning and does not load Drives branding: the widget fetches {base}/api/widget-branding?org={slug} (logo, org name, primary color, app name) and opens the chat page at {base}/widget/chat?org={slug}. Different slugs on different pages serve differently-branded widgets from the same deployment.
data-base No Origin URL The origin the script was loaded from (new URL(script.src).origin) Selects the Veru365 environment the widget talks to. Because the default is the script's own origin, a tag whose src already points at the right environment needs no data-base at all — the common case is just src + data-org.

Behavior guarantees baked into the script: self-contained vanilla-JS IIFE — no framework, no leaked globals; all DOM ids and classes are prefixed v365w- and its styles are scoped, so host-page CSS and widget CSS cannot collide. If the branding fetch fails, the bubble falls back to a neutral default color (#6366f1) rather than erroring. The applied primary_color is validated as a hex color before use.

Honest scope note: today data-org drives branding only. In this demo environment the underlying chat runs against a shared demo dataset regardless of which org slug embeds the widget. Per-org anonymous/lead chat context is a pilot/roadmap item — see "Demo scope vs production hardening" below.
What the widget renders

Streaming text plus real financial modules, inline in chat

This is not a text-only bot. Responses stream token-by-token over SSE (server-sent events), and when the assistant runs a planning tool, the result arrives as a rich interactive module rendered directly inside the chat — the same components the full advisor platform uses.

Inline modules

The three modules this demo showcases:

  • Monte Carlo retirement projection — an interactive probability-of-success chart (the platform's real Monte Carlo chart component, not a screenshot).
  • Net-worth card — assets, liabilities, and net worth at a glance.
  • What-if comparison — side-by-side scenario outcomes (e.g. retiring at 60 vs 65).

Technically: card-bearing tool results arrive as modal_serve SSE events carrying a backend-resolved component name; the widget reuses the platform's shared inline-card renderer, so the widget and the signed-in portal render from one component vocabulary (net-worth, financial plan, Monte Carlo result, what-if comparison, portfolio summary, stock analysis, budget overview, tax profile, and more). Events that only make sense inside the full app shell (navigation, side panels, canvas) are ignored cleanly by the widget.

Panel geometry and expand behavior

  • Launcher: a 56×56px bubble fixed bottom-right (20px offsets), tinted with the org's primary color, with accessible open/close labels.
  • Docked panel: 420×600px iframe, bottom-right, rounded corners, capped to the viewport so it never overflows small windows.
  • Expanded panel: a maximize button in the chat header grows the panel to min(720px, calc(100vw - 40px)) wide — room for charts to breathe. The expanded/compact choice survives closing and reopening the panel.
  • Chart expand: chart-bearing cards (Monte Carlo, what-if, financial plan, stock analysis, portfolio summary) additionally get an Expand button that opens the module in an overlay dialog inside the panel — closable by backdrop click, X, or Escape.
  • Mobile (≤480px): the panel goes full-viewport automatically. No extra configuration for mobile templates.
Conversation features

Chips in, leads out

The widget is built to move an anonymous visitor from curiosity to a captured lead in the advisor's inbox — without a login.

Quick-reply chips

First-time visitors see four starter prompts that are verified to produce rich-module answers:

  • "Am I on track for retirement?"
  • "What's my net worth picture?"
  • "Generate my full financial plan with Monte Carlo simulations"
  • "What if I retired at 60 instead of 65?"

Between turns, up to three follow-up chips appear (never while a response is streaming), and chips the visitor already used that session are filtered out.

Lead capture — how a widget conversation reaches the client's advisor

  • Show value first: the lead form appears exactly once per session, and only after the first response that rendered at least one rich module.
  • Availability preflight: before rendering the form, the widget checks whether the org has published a public advisor profile (backed by the public GET /api/public/advisor/{slug} endpoint). Orgs that haven't published get a plain "reach out" message instead of a form that could only fail on submit.
  • Submit: name + email (+ optional message) forward to the public, tenant-scoped lead endpoint POST /api/public/advisor/{slug}/lead. Nothing is echoed back to the visitor; the lead is stored in the client org's tenant.
  • Advisor side: the client firm's advisors read captured leads through the authenticated GET /api/advisor/leads API and portal — lead delivery is pull, not push, today. (Orgs can separately register outbound webhooks — an org-owner action — for platform events like user.created, fact_finder.completed, and branding.updated; a dedicated lead event is not yet in that catalog.)
  • Rate limiting: lead submission is rate-limited at 5/min per IP on the backend; the widget shows a graceful "try again in a minute" message on 429.
How it works under the hood

Why a third-party origin like github.io works today

This page itself is served from GitHub Pages — a domain Veru365 doesn't control — and the widget runs fine on it. Here is the cross-origin model that makes that true, in plain terms first: the widget lives in its own sealed window inside your page, talks only to Veru365, and can neither read your page nor be broken by it.

Iframe isolation

The bubble and panel shell are injected by widget.js; the chat itself renders inside an iframe pointed at the Veru365-hosted page /widget/chat?org={slug}. That page is a standalone chat surface — no app shell, no sidebar, no login flow. Host page and chat are separate browsing contexts: your scripts and CSS can't reach into the chat, and the chat can't reach into your page. The only host-page communication is a size-toggle postMessage (expand/collapse), and the script verifies the message actually comes from its own iframe before acting on it.

Frame-ancestors: embeddable widget, non-embeddable app

The Veru365 app as a whole refuses to be framed by third parties (X-Frame-Options: SAMEORIGIN + CSP frame-ancestors 'self' on every route). Exactly three surfaces carve out of that default so host pages can frame them: /widget, /widget/*, and /widget.js. The route matching is deliberately precise so no future route can become embeddable by accident. In the current demo environment the carve-out is frame-ancestors * — any site can embed — which is explicitly demo scope; production replaces * with a per-org domain allowlist.

One origin, three proxies

The widget only ever talks to the Veru365 app origin. Three small server-side proxies sit behind it:

  • /api/widget-branding — proxies the backend's public, no-auth branding-by-slug endpoint, with CORS enabled (branding is public data) and a 60-second cache. This is why the branding fetch works from any origin.
  • /api/widget-chat — streams chat over SSE. Credentials are injected server-side; no token ever reaches the browser or the host page. The request body is allowlisted (message, session id, conversation id) so a host page cannot smuggle arbitrary fields upstream.
  • /api/widget-lead — availability preflight + lead submission, passing through the backend's public lead endpoints.

In this demo environment, every one of these proxies is dark unless explicitly enabled per environment via server configuration — production is structurally unaffected by the demo setup.

For template builders

Page-integration notes

Practical answers for putting the tag into landing-page templates at scale.

  • Load order: anywhere in the document works; end of <body> with defer is the recommended pattern. The script waits for DOM readiness and the heavy part (the iframe) doesn't load until first open.
  • Footprint: one 56px bubble at bottom-right with 20px offsets, at z-index 2147483000 (above virtually everything). If your template has its own bottom-right floating element (a cookie bar, a "back to top" button), place it elsewhere or accept the widget on top.
  • Mobile: nothing to configure — at ≤480px the open panel takes the full viewport and returns to the page on close.
  • Many pages, one org: every page of a client's web presence uses the identical tag with the same data-org. There is no per-page registration, key, or dashboard step — the demo hub and the three "Acme Wealth" landing pages each carry the identical snippet, copy-pasted.
  • Many clients, one template: the only thing that varies per client is the data-org slug, so a template engine can inject it as a single variable — the Summit Planning demo page is exactly that: the same snippet with a different slug, serving a second brand.
  • Sessions: each page load is an independent chat session with multi-turn context within that session. Sessions are page-scoped in the current build; cross-page conversation continuity is not yet supported.
  • Attribution: the chat footer shows "Powered by {app name}". Removing it is planned as a reseller-tier entitlement, not available today.

If your pages ship a Content-Security-Policy

Landing-page templates often carry a strict CSP. The widget needs three allowances for the Veru365 origin your src points at (call it <veru365-origin>):

  • script-src <veru365-origin> — loads widget.js itself.
  • connect-src <veru365-origin> — the script fetches the org's branding (colors, name) from the host page context.
  • frame-src <veru365-origin> — the chat panel is an iframe served from that origin. Everything the chat does (SSE streaming, lead submission) happens inside the iframe, so no further host-page directives are needed.

Payload and updates: widget.js is small — about 7 KB raw, roughly 3 KB compressed — and the heavy part (the chat iframe) doesn't load until first open. The tag is unversioned by design: widget updates ship with our deployment and already-embedded pages pick them up on their next page load, with no re-tagging across a template fleet.

Being transparent

Demo scope vs production hardening

Everything demonstrated on this site is real and running. These are the items where the demo configuration is deliberately looser than a production rollout would be, stated plainly.

CapabilityStatus todayProduction plan
Embed from any domain Available today — frame-ancestors * and open branding CORS mean any site can embed any org's widget Pilot — per-org domain allowlist: each client org declares which domains may embed its widget; provisioned together during pilot
Per-org widget branding Available today — data-org switches the widget's branding per client org, live-verifiable on this site's two demo brands Unchanged — same mechanism in production
Per-org chat identity Roadmap — not available today: chat runs against a shared demo dataset regardless of which org embeds the widget Roadmap — anonymous/lead session auth with true per-org scoping, so each client's widget conversations are their own
Lead capture & rate limiting Available today — live end-to-end; the backend limit (5/min/IP) is currently measured at the proxy's egress IP, so all widget visitors share one bucket — fine at demo scale Pilot — forwarding real client IPs for per-visitor limits at reseller traffic volumes
White-label attribution Available today — footer reads "Powered by {your client's app name}" when the org sets one Roadmap — removing the footer entirely as a reseller-tier entitlement
Partner credentials Pilot — org provisioning and platform credentials are issued per-partner by Veru365 during a pilot, not self-serve Roadmap — self-serve API keys

Why we list this at all

You're evaluating a platform your clients will put their brand on. We'd rather you learn the current edges from us than discover them in diligence. Everything marked "Available today" is verifiable against the running demo on this site; "Pilot" items are configured with you during onboarding; "Roadmap" items don't exist yet and we won't pretend they do.

Troubleshooting

When something doesn't show up

The three failure modes you'll actually hit, with causes and checks.

The bubble doesn't appear

  • Missing data-org: the widget logs a console warning and refuses to load. Check the browser console first.
  • Invalid slug: slugs must match ^[a-z0-9][a-z0-9-]{0,63}$ — lowercase, no spaces, no leading/trailing hyphen.
  • Script blocked: verify the src URL loads directly in a browser tab, and that no ad blocker or CSP on the host page is blocking third-party scripts.
  • Two tags on one page: harmless — the second is intentionally a no-op. Not a cause of missing bubbles, but worth knowing when templates compose.

Branding doesn't load (bubble is generic indigo)

  • This is the designed fallback: if the branding fetch fails or returns nothing usable, the widget stays functional with a neutral default color rather than breaking.
  • Check the slug: open {base}/api/widget-branding?org={slug} directly — it should return the org's name and colors. An unknown slug returns an error and the widget falls back.
  • Check the color value: the org's primary color is only applied if it's a valid hex color; anything else is ignored defensively.

A rich module doesn't render

  • Not every answer carries a module: modules render when the assistant runs a card-producing planning tool. The four quick-reply chips are verified to produce at least one module each; free-form questions may come back as text only.
  • Known-bad prompts in the current build: stock-analysis questions route to a part of the assistant not yet reachable inside the widget, and market-crash stress-test prompts currently hit a known bug we're fixing — both are issues in the current build, not embed problems.
  • Raw markup in long answers: long-form answers (notably the full financial-plan prompt) may occasionally show raw value-source markup in the prose in the current demo build — a rendering fix is in review.
  • Red inline error bubble: the demo backend can cold-start slowly; stream errors surface as a graceful inline message ("Sorry, the chat service is unreachable."). Retry the message — the widget and your page are unaffected.
