Provisioning API Reference — Onboard a Client in Four Calls

The overview page shows the four-call flow at a glance. This page is the engineer's version: the auth model, each endpoint's real request and response shapes, the events that flow back to your systems, and where the OpenAPI schema lives so you can generate typed clients. Everything marked Available today is live and verifiable; anything we'd set up with you is labeled Pilot, and anything that doesn't exist yet is labeled Roadmap — no exceptions.

Section 1

Authentication and the partner model

In plain terms: there are two hats. Your partner token creates and manages client organizations from the outside. Each client's org-owner token controls what happens inside that organization — its branding, its people, its webhooks. Keeping those separate is deliberate: it means we can never quietly rewrite a client's brand or membership on their behalf, and neither can you without being the actual owner of record.

Provisioned together during pilot

The partner token (platform admin)

All authentication is Auth0-issued JWTs sent as Authorization: Bearer <token>. The partner token is a JWT for an account with platform-admin (superuser) privileges. It can call everything under /api/platform-admin/*: create orgs, set status and trial clocks, flip feature switches, add members, send org-setup invites, and read cross-org billing reports.

Honest note on credentials: there is no self-serve signup for partner credentials today. During a pilot, Veru365 provisions the partner account with you. The only self-serve API-key surface in the product is an org-scoped key for the inbound integration webhook receiver (POST /api/integrations/keys, scope webhook:write) — it authorizes exactly one endpoint and nothing else. Self-serve partner keys are Roadmap.

Available today

The org-owner token

An org-owner token is an ordinary member JWT where the caller's role in the org is owner. It controls branding, invitations, members, the shareable invite link, webhook endpoints, and org settings.

A guardrail worth knowing about: writes to branding, webhooks, invite links, and settings are protected by an impersonation block. A platform superadmin who is not a real member of the org is audit-logged and 403-blocked from these writes — the write must come from the org owner's own JWT. If you want your operations account to manage a client org directly, add it as a real owner first via POST /api/platform-admin/orgs/{org_id}/members (that action is audited too).

Available today

Org context travels in a header, not the URL

For all /api/orgs/{org_id}/... endpoints, the active org is resolved from an X-Org-Id (numeric) or X-Org-Slug header — the {org_id} path segment is largely vestigial. Always send the header:

curl -X GET "https://api.veru365.com/api/orgs/47/members" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -H "X-Org-Id: 47"
Auth classMechanismCan touch
publicnone (some per-IP rate limits)Branding reads by slug, invitation preview, lead capture, public advisor page

Section 2 · Call one

Create an organization

One call provisions an isolated tenant. "Isolated tenant" is not marketing: every client org's data is separated at the database layer with row-level security, so one client's advisors, end-clients, conversations, documents, and financial data are invisible to every other org — including your other clients on the same deployment.

Available today

POST/api/platform-admin/orgs

Gotcha, stated up front: this endpoint takes query parameters, not a JSON body. Typed clients generated from the OpenAPI schema get this right automatically; hand-written calls frequently don't.

Query paramTypeNotes
namestring, requiredOrg display name; slug derived. 409 on name/slug conflict.
owner_user_idint, requiredThe user must already exist. To onboard a brand-new owner, use an org-setup invite instead (below).
trial_daysint > 0, optionalOrg is created in trial status with trial_expires_at = now + trial_days.
curl -X POST "https://api.veru365.com/api/platform-admin/orgs?name=Acme%20Wealth&owner_user_id=1042&trial_days=30" \
  -H "Authorization: Bearer $PARTNER_TOKEN"

# 201 →
{"id": 47, "name": "Acme Wealth", "slug": "acme-wealth", "status": "trial",
 "trial_expires_at": "2026-08-08T00:00:00Z"}

Don't have a user yet? Use an org-setup invite

When the client's owner doesn't have an account, send an org-setup invite. It emails an onboarding link; when the recipient accepts and completes setup, they create their own org and become its owner:

curl -X POST "https://api.veru365.com/api/platform-admin/org-setup-invites" \
  -H "Authorization: Bearer $PARTNER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"email": "jordan@acmewealth.com", "first_name": "Jordan"}'

# 201 → includes token, invite_url, expires_at (7 days), idv_required

Lifecycle after creation

  • PATCH /api/platform-admin/orgs/{org_id} — query params status and/or trial_expires_at. Enforced transitions: active→suspended, trial→active|suspended, suspended→active.
  • POST .../{org_id}/suspend and POST .../{org_id}/activate — convenience state flips.
  • Ending a relationship is a cascade, not a status flip. Setting status=deactivated directly is rejected with 400. Use POST .../{org_id}/deprovision: it deactivates the org, soft-removes every member, revokes live bank connections, and re-homes orphaned users. Nothing is hard-deleted — all rows are retained for compliance.
  • GET /api/platform-admin/orgs — searchable, filterable list with member counts, conversation counts, token usage, trial countdown, and onboarding progress per org. GET .../orgs/{org_id}/health gives a deep per-org health view. CSV export exists.
Section 3 · Call two

Apply branding

Colors and product name go in as JSON; logo and favicon upload as files. Branding applies across the client's portal, their login page, and the embedded chat widget — the widget reads branding through a public endpoint keyed by the org slug, cached for 60 seconds, so a change is visitor-visible within about a minute.

Available today

PATCH/api/orgs/{org_id}/branding

Auth: org owner's own JWT (superadmin impersonation is 403-blocked — see Section 1). All fields optional, but an empty body is a 400.

FieldValidation
app_name, display_namedisplay_name ≤ 80 chars. This is the product name end users see — the browser tab says the client's name, not ours.
primary_color, secondary_color, accent_colorprimary_color/accent_color must be #RRGGBB hex.
logo_url, favicon_urllogo_url must be https:// and ≤ 512 chars.
custom_css≤ 4096 chars; must not contain <script.

Validation failures return 422 with a {"validation_errors": [...]} list. A successful write also fires a branding.updated webhook to your endpoints (it reports which fields changed, never the values).

curl -X PATCH "https://api.veru365.com/api/orgs/47/branding" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -H "X-Org-Id: 47" \
  -H "Content-Type: application/json" \
  -d '{"app_name": "Acme Wealth", "primary_color": "#0E7C5A", "secondary_color": "#0A3D2E"}'

Logo and favicon upload

POST/api/orgs/{org_id}/branding/logo and .../branding/favicon — multipart form, field file, image-validated, 5 MB cap. The response gives you a public proxy URL (file_url) that renders on unauthenticated pages — that's the URL you store back into branding.

curl -X POST "https://api.veru365.com/api/orgs/47/branding/logo" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -H "X-Org-Id: 47" \
  -F "file=@acme-logo.png"

# → {"file_url": "https://api.veru365.com/api/public/branding/…", "file_key": "branding/…"}

Reading branding back (no auth required)

  • GET /api/orgs/url-handle/{slug}/branding — public; the same endpoint the widget and login page use. Returns {org_name, slug, logo_url, primary_color, secondary_color, favicon_url, app_name}.
  • GET /api/orgs/{org_id}/branding/preview — member; branding rendered as CSS variables, handy for building your own preview UI.
Section 4 · Call three

Set feature switches

Every org carries a set of on/off switches for platform capabilities — bank connections, document vault, portfolio tools, tax, budgeting, scheduling, CRM integrations. If you sell tiers ("Starter" without bank sync, "Premium" with everything), this is the endpoint your packaging maps onto: one PATCH per client at provisioning time, or later when they upgrade.

Available today

PATCH/api/platform-admin/orgs/{org_id}/features

Auth: partner token. Body is a JSON object under a features key; values merge into the org's existing flags. Unknown keys are rejected with 400 — the flag list is canonical, so a typo fails loudly instead of silently doing nothing.

curl -X PATCH "https://api.veru365.com/api/platform-admin/orgs/47/features" \
  -H "Authorization: Bearer $PARTNER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"features": {"enable_plaid": true, "enable_file_vault": true, "enable_scheduling": true}}'

# → {"org_id": 47, "features": { …effective flags after merge… }}
Flags (default ON)What it gates
enable_plaidBank/brokerage account connections
enable_file_vaultCompliant document vault
enable_portfolioPortfolio analysis tools
enable_tax / enable_budgeting / enable_pfsTax profile, budgeting, personal financial statements
enable_scheduling / enable_calendly / enable_google_calendar / enable_microsoftAppointment scheduling and calendar integrations
enable_wealthbox / enable_salesforce / enable_hubspot / enable_ringcentralCRM and call-log sync
require_identity_verificationWhether members must pass identity verification

A further set of experimental flags (agent memory, agentic workflows, and similar) exists and defaults to OFF; those are flipped by Veru365 platform staff rather than partners.

Any member of the org can read the effective flags: GET /api/orgs/{org_id}/features returns the flat dict of defaults merged with the org's overrides — useful if your own UI needs to know what a given client has.

Section 5 · Call four

Invite the advisor

The invitation call returns a ready-to-send link, so your onboarding flow works even if the invite email bounces. Role model in one paragraph: an org has owners (full control of the org: branding, members, webhooks, settings) and users (the advisor's clients — they use the platform, they don't administer it). There's also a middle "Manager" option: a user-role member granted restricted admin powers, who can invite and manage plain user-role clients but only sees invitations they themselves sent and can never mint owners.

Available today

POST/api/orgs/{org_id}/invitations

Auth: org owner (or a restricted admin, for user-role invites only).

Body fieldNotes
emailRequired.
role"owner" or "user" (default "user").
first_name, last_name, job_titleOptional.
idv_requiredDefault false — require identity verification before the invite can be accepted.
grant_restricted_adminDefault false — the "Manager" invite described above.
curl -X POST "https://api.veru365.com/api/orgs/47/invitations" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -H "X-Org-Id: 47" \
  -H "Content-Type: application/json" \
  -d '{"email": "advisor@acmewealth.com", "role": "owner", "first_name": "Jordan"}'

# 201 → full invitation row incl. token, expires_at, and:
#   "email_delivery": {"sent": true, "error": null}
  • Invitations expire in 7 days. POST .../invitations/{id}/resend re-sends; DELETE soft-cancels (rows are never hard-deleted; accepted invitations can't be cancelled).
  • Check email_delivery.sent. The invitation row is created even if the email fails, so you can always share the invite URL through your own channel.
  • 409 if the person is already a member or already has a pending invite.

Onboarding at reseller scale

  • Available today POST /api/orgs/{org_id}/invitations/bulk — up to 100 invitations per request with partial-success semantics: the response separates created from per-item errors, so one bad email doesn't abort the batch.
  • Available today GET /api/orgs/{org_id}/invite-link — a shareable, unlimited-use invite code (…/welcome?code=XXXXXXXX), lazily created on first read, rotatable with a POST. Good for "put one signup link on the client's site" flows. (Org owner only — restricted admins can't use the invite link.)
Section 6

Events out: webhooks and lead capture

Provisioning is calls in; this is data flowing back. Each client org can register webhook endpoints that push signed events into your systems, and each client gets a public, rate-limited lead-capture endpoint plus a bookable public advisor page — so a widget conversation on a landing page can end as a lead record and a booked meeting.

Available today

POST/api/orgs/{org_id}/webhooks/endpoints

Auth: org owner. Max 10 endpoints per org. Body:

curl -X POST "https://api.veru365.com/api/orgs/47/webhooks/endpoints" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -H "X-Org-Id: 47" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://hooks.yourplatform.com/veru365",
       "secret": "whsec_yourrandomsecret",
       "events": ["fact_finder.completed", "pago.*"]}'

The events filter (default ["*"]) accepts exact event names or glob-prefix patterns like "pago.*". Every delivery is a POST with this envelope and headers:

{"event": "fact_finder.completed", "occurred_at": "2026-07-09T18:31:04Z",
 "org_id": 47, "data": { … }}

# Headers on every delivery:
Content-Type: application/json
X-Webhook-Signature: sha256=<hex HMAC-SHA256 of the raw body, keyed by your endpoint secret>
User-Agent: Veru365-Webhooks/1.0
  • Verify the signature. The secret is optional but strongly recommended — without one, X-Webhook-Signature is empty and deliveries are unauthenticated.
  • Retries: 3 attempts total (delays 2s, then 8s), 10-second timeout, best-effort — failures are logged and never block the platform.
  • Delivery audit built in: GET .../webhooks/endpoints returns per-endpoint stats (last_delivered_at, last_status_code, total_deliveries, total_failures).

Current event catalog

Machine-readable at GET /api/integrations/webhooks/events (any authenticated member). Today's catalog: user.created, fact_finder.completed, pago.score_updated, branding.updated, org.trial_expired. A dedicated lead event (e.g. lead.created) is Roadmap — today you pull leads via the authenticated GET /api/advisor/leads endpoint, so nothing is lost, it's just pull rather than push.

Available today

Public lead capture and the bookable advisor page

GET/api/public/advisor/{slug} — no auth. Returns the org's public advisor card (name, firm, scheduling URL, branding) for the public booking page. It 404s unless the org has explicitly published its advisor (a public_enabled flag plus a scheduling URL in org settings) — unknown and unpublished slugs are deliberately indistinguishable, and the advisor's phone number is intentionally omitted from the public payload.

POST/api/public/advisor/{slug}/lead — no auth, rate-limited 5/min per IP. Body: {"name": "…", "email": "…", "message": "…?"} (name ≤ 200 chars, message ≤ 2000). Returns {"ok": true} and nothing else — it never echoes data back. Leads land tenant-scoped in the client's org; advisors read them via the authenticated GET /api/advisor/leads.

This is the same pair of endpoints the embedded chat widget's lead-capture card uses — the form only renders for orgs whose advisor page is published, so visitors never see a form that can only fail.

Available today

Inbound events (your systems → Veru365)

The reverse direction exists too: POST /api/integrations/webhook accepts custodian/CRM events, authenticated by an org-scoped X-API-Key. Keys are created by an org admin at POST /api/integrations/keys — the raw key is returned exactly once; only its SHA-256 hash is stored. This key authorizes only this receiver.

Section 7

End-to-end example: "Acme Wealth," zero to live widget

One annotated script matching the overview page's four steps. It assumes the partner token exists (pilot-provisioned) and the owner accepted their org-setup invite — or already had an account, in which case step 0 is skipped.

#!/usr/bin/env bash
# Provision "Acme Wealth" end to end.
# PARTNER_TOKEN  — pilot-provisioned platform-admin JWT
# ORG_OWNER_TOKEN — the client owner's JWT (their account, their writes)
BASE="https://api.veru365.com"

# 0. (If the owner has no account yet) send an org-setup invite and stop here.
#    On acceptance the owner creates their own org and becomes its owner — so
#    SKIP step 1 in that path and look up the new org's id/slug via
#    GET /api/platform-admin/orgs instead. Step 1 applies only when the owner
#    already had an account (no setup invite sent).
curl -X POST "$BASE/api/platform-admin/org-setup-invites" \
  -H "Authorization: Bearer $PARTNER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"email": "jordan@acmewealth.com", "first_name": "Jordan"}'

# 1. Create the org — NOTE: query params, not a JSON body. Capture the id + slug.
ORG_JSON=$(curl -s -X POST \
  "$BASE/api/platform-admin/orgs?name=Acme%20Wealth&owner_user_id=1042&trial_days=30" \
  -H "Authorization: Bearer $PARTNER_TOKEN")
ORG_ID=$(echo "$ORG_JSON" | jq -r .id)      # → 47
ORG_SLUG=$(echo "$ORG_JSON" | jq -r .slug)  # → acme-wealth

# 2. Branding — owner token + org-context header. Upload the logo first,
#    then PATCH colors/name (logo file_url from the upload response).
LOGO_URL=$(curl -s -X POST "$BASE/api/orgs/$ORG_ID/branding/logo" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" -H "X-Org-Id: $ORG_ID" \
  -F "file=@acme-logo.png" | jq -r .file_url)

curl -X PATCH "$BASE/api/orgs/$ORG_ID/branding" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" -H "X-Org-Id: $ORG_ID" \
  -H "Content-Type: application/json" \
  -d "{\"app_name\": \"Acme Wealth\", \"primary_color\": \"#0E7C5A\",
       \"secondary_color\": \"#0A3D2E\", \"logo_url\": \"$LOGO_URL\"}"

# 3. Feature switches — package the tier this client bought.
curl -X PATCH "$BASE/api/platform-admin/orgs/$ORG_ID/features" \
  -H "Authorization: Bearer $PARTNER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"features": {"enable_plaid": true, "enable_file_vault": true,
                    "enable_scheduling": true}}'

# 4. Invite the advisor team — check email_delivery.sent; the invite URL is
#    in the response either way. Bulk endpoint takes up to 100 at once.
curl -X POST "$BASE/api/orgs/$ORG_ID/invitations" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" -H "X-Org-Id: $ORG_ID" \
  -H "Content-Type: application/json" \
  -d '{"email": "advisor@acmewealth.com", "role": "owner", "first_name": "Jordan"}'

# 5. Wire events back to your platform (HMAC-signed).
curl -X POST "$BASE/api/orgs/$ORG_ID/webhooks/endpoints" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" -H "X-Org-Id: $ORG_ID" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://hooks.yourplatform.com/veru365",
       "secret": "whsec_yourrandomsecret", "events": ["*"]}'

# 6. The widget — one line on any page the client owns, any number of pages:
#    <script src="https://app.veru365.com/widget.js" data-org="acme-wealth" defer></script>
Demo-environment note: the live widget on this demo site runs against our staging environment, and the widget embed itself is a staging demo capability today. Per-org anonymous chat scoping and per-domain embed allowlisting are Roadmap — they don't exist yet and we won't pretend they do. The provisioning endpoints above, however, are the production API surface.
Section 8

OpenAPI 3 schema and client generation

You shouldn't hand-write any of the calls above. The full API ships a complete OpenAPI 3 schema — several hundred documented paths — from which you can generate typed clients in any language.

Available today

Where the schema lives

  • GET /openapi.json on the API base — the complete machine-readable schema (the same schema we generate our own typed clients from).
  • Public docs endpoints, no auth: GET /api/docs/ (index), /api/docs/integration-spec, /api/docs/webhooks, /api/docs/auth-guide.
# TypeScript
npx openapi-typescript https://api.veru365.com/openapi.json -o veru365.d.ts

# Python / Java / Go / anything
openapi-generator generate -i https://api.veru365.com/openapi.json \
  -g python -o ./veru365-client

Error shape

  • 400 — malformed request or disallowed operation (empty branding body, unknown feature flag, forbidden status transition).
  • 403 — authenticated but not allowed (impersonation block, restricted-admin role escalation).
  • 404 / 409 / 410 — not found (or deliberately hidden, on public endpoints); conflict (duplicate org name, duplicate pending invite, already a member); expired (invitation tokens).
  • 422 — validation failure; branding returns a structured {"validation_errors": [...]} list, date params return standard validation errors.
  • 429 — rate limits on the public endpoints (lead capture 5/min/IP, invitation preview 10/min/IP).

Idempotency and retry notes

  • There is no idempotency-key header today (Roadmap). In practice the natural keys guard you: re-running org creation with the same name returns 409, re-inviting the same email returns 409 — so a blind retry of the provisioning script fails loudly rather than duplicating.
  • PATCHes (branding, features) are safe to retry — they merge the same values again.
  • Webhook deliveries are at-least-once in effect (3 attempts) — make your receiver idempotent on event + occurred_at + org_id.
  • On transient 5xx/timeout, retry with backoff; on 409, treat as "already done" and read the resource instead.
Section 9

The no-code alternative

The core of this page — creating orgs, feature switches, branding, and invitations — is also available point-and-click in the Veru365 admin console, which hits the same backend with the same permission rules. Plenty of teams run their first handful of clients from the console and only reach for the API once volume makes automation worth it; for anything the console doesn't yet surface, the API above is the complete interface.

A sensible pilot shape: we provision your partner account, onboard the first client together in the console so your team sees every screen, then hand you the script from Section 7 as the template for automating the rest.
