Architecture, Security & Compliance — built for a regulated industry

How Veru365 keeps every client firm's data isolated, recorded, and auditable — explained in plain English first, with the technical detail underneath.

The one-paragraph version

Every client firm is an isolated tenant — and the isolation lives in the database, not just the app

Each firm you deploy Veru365 for (in our world, an "organization") gets its own walled-off slice of the platform. What makes that claim worth more than marketing copy: the wall is enforced by PostgreSQL row-level security inside the database itself. Every query the application runs is automatically scoped to exactly one organization at the database layer, so even a bug in application code cannot read or write another firm's rows. On top of that sits soft-delete record-keeping, audit logging of any platform-staff access, and GDPR export/erasure tooling — the things an advisor's compliance team asks about on day one.

System at a glance

What talks to what

A visitor on any of your clients' web pages sees one chat bubble. Everything behind it runs in our cloud — the host page carries a single script tag and nothing else.

Three things worth noticing in that picture:

  • The host page is inert. The widget is a self-contained script that lazy-loads an iframe served from our origin. It shares no cookies, no globals, and no styles with the page it sits on, and every network call it makes is wrapped so a failure can never break your client's website.
  • The browser only ever talks to one origin — ours. Chat, branding, and lead-capture requests all go through our application server, which holds any credentials server-side. No API tokens or secrets are ever present in the visitor's browser.
  • The AI sits behind the API, not in front of it. Model calls run through Google Cloud's Vertex AI from inside our cloud environment; the model reaches financial data only through the API's controlled tool calls, never by querying the database itself.

Everything runs on Google Cloud in the United States (region us-central1) — Cloud Run services in front of a private-network PostgreSQL instance that accepts no public connections.

Tenant isolation, honestly explained

What "row-level security" actually buys you

Available today

Plain English: most multi-tenant software keeps every customer's data in the same tables and relies on the application code remembering to add "…and only for this customer" to every query. That works until one developer forgets once. Veru365 moves that guarantee into PostgreSQL itself: the database is told, per request, which organization is active, and row-level security policies on every tenant-scoped table make rows belonging to other organizations invisible — reads and writes both.

Technically: each API request resolves the caller's organization from their authenticated identity plus an explicit org-context header, and binds it to the database session before any query runs. The RLS policies then filter every statement by that organization ID. The failure mode this changes is the important part:

  • A bug in application code that omits a tenant filter returns zero rows from other firms — not a data leak, an empty result.
  • An insert or update aimed at the wrong tenant is rejected by the database, not silently written.
  • Cross-tenant access paths (the platform-staff console) are explicit, narrow, and audited — see the records section below.

We treat this as the load-bearing wall of the platform: changes touching tenant-scoped data are gated before any production release by integration tests that run against a real PostgreSQL instance, specifically because SQLite-style unit tests cannot exercise RLS.

Why this matters to you as a reseller: when two of your clients — even direct competitors — run on the same deployment, the promise that neither can ever see the other's client data does not depend on our developers being careful. It's enforced by the database engine on every single query.

Records & retention

A record trail an advisor's compliance officer can live with

Advisors are regulated communicators. If an AI assistant discusses someone's retirement with them, that conversation is a business record — so the platform is built record-first.

Available today

What gets recorded

  • Conversations are retained, tenant-scoped, like any other advisor communication — including the tool calls and computed figures behind each answer, not just the prose.
  • Delete means deactivate, never erase the trail. This is a deliberate, platform-wide pattern: cancelling an invitation soft-cancels the row rather than deleting it; offboarding an entire firm (deprovisioning) deactivates the organization, disconnects live bank/brokerage links, and re-homes users — while retaining every row. There is no "hard delete an org" button, on purpose. (The regulated exception: verified GDPR erasure requests, below.)
  • Platform-staff access is logged and constrained. When Veru365 staff view a tenant for support, that access is written to a superadmin access log your diligence team can ask about — and staff who are not real members of the firm are blocked at the API layer from making changes through the firm's own admin surfaces (branding, settings, webhooks). Platform-level lifecycle actions, such as adding a member, go through a separate superadmin surface where every action is audit-logged. Support can look with an audit trail; it cannot quietly touch.
  • Outbound webhook deliveries carry HMAC-SHA256 signatures when a signing secret is configured on the endpoint (recommended, and part of pilot setup), plus per-endpoint delivery/failure counters, so event feeds into your systems are verifiable and accounted for.
Configured together during pilot

Long-horizon archive retention

Chat-archive storage with an enforced multi-year retention policy is built into the platform. Certified write-once (WORM) locking and the archive pipeline are enabled together with each partner during pilot diligence rather than switched on generically — locking is irreversible, and the right retention regime (17a-4-style WORM versus advisers-act books-and-records) depends on which rules your clients answer to. Retention windows, legal-hold behavior, and regulator-facing export formats are configured in the same conversation.

Privacy & data rights

What we collect, where it lives, and how it leaves

Available today

The public widget collects almost nothing

A visitor chatting on a public landing page is anonymous: no login, no account, a random per-page-load session ID. Personal data enters only if the visitor chooses to submit the lead form — name, email, and an optional message — which is stored tenant-scoped for that firm's advisors and never echoed back in the API response. The lead endpoint is rate-limited per IP to blunt scraping and spam. The signed-in portal is the opposite end of the spectrum — full financial-planning data — and everything there sits behind Auth0-issued identity plus the tenant isolation described above.

Available today

GDPR-style export and erasure

Data-subject tooling is built and running: export and erasure workflows execute as scheduled jobs in production. Erasure is the one sanctioned exception to the "never hard-delete" posture — a verified data-subject request removes personal data through a controlled workflow, rather than an engineer running ad-hoc SQL.

Available today

Data residency

All services and data run on Google Cloud in the United States (us-central1). The production database is reachable only over the private network — it has no public IP. Uploaded brand assets (logos, favicons) live in a private storage bucket and are served through an allowlisted proxy, not exposed directly.

AI safety posture

The assistant computes; it doesn't improvise numbers

Available today

The design principle: when the assistant shows a retirement projection, that Monte Carlo simulation actually ran. Dollar figures come from tool calls into the same planning engine the full platform uses — net-worth calculations, plan projections, what-if scenarios — and are rendered as structured, interactive modules in the chat rather than free-form model prose. The interesting numbers on screen are computation output, not model output.

  • Figures carry provenance. Sourceable dollar amounts are emitted with machine-readable source tags identifying the computation they came from, so a number in a transcript can be traced back to its origin.
  • The model has no database access. It reaches data exclusively through the API's tools, inside the same per-tenant scoping as every other request.
  • Conversations are retained and auditable like any advisor communication — the records section above applies in full to AI output.
  • Quality is tested, not assumed: we run evaluation suites against the assistant's tool-use behavior as part of development, and model changes are staged and human-gated before production.
What we won't claim yet

The honest edges

Read this section before your security team does

  • No SOC 2 stamp — and we won't pretend otherwise. We do not currently claim SOC 2 or ISO certification. Formal certifications, penetration-test summaries, and your security questionnaire are handled during pilot diligence, where we can show rather than assert.
  • The public demo widget runs in demo scope. Today, any site can technically embed the demo widget, and the demo chat runs against shared demonstration data regardless of which brand is shown. The production widget plan adds a per-firm domain allowlist (so only approved sites can embed a firm's widget) and per-firm anonymous visitor scoping. We'd rather tell you that here than have you find it in a browser console.
  • Billing is reporting, not invoicing. Per-org usage metering and billing reports are live; invoicing is currently a manual process, not a self-serve payment flow.
  • Partner provisioning is done with us, not self-serve. Creating firm tenants uses a platform-admin API operated with Veru365 during pilot — there is deliberately no self-serve credential that grants platform-wide access.
Diligence starters

Questions we invite you to grill us on

Short honest answers here; long-form answers, with evidence, during pilot diligence.

If a developer ships a buggy query, can Firm A see Firm B's clients?

No — row-level security in PostgreSQL filters every query by the active organization, so an unscoped query returns nothing from other tenants rather than leaking them.

Can your own staff read or edit our clients' data?

Staff support access exists, is written to an access log, and staff who aren't real members of a firm are blocked at the API from making changes through the firm's own admin surfaces; platform-level lifecycle actions run through a separate, fully audit-logged superadmin surface. Look with a trail: yes. Touch silently: no.

What personal data does the widget collect from an anonymous visitor?

None automatically. Name, email, and an optional message — only if the visitor submits the lead form, stored tenant-scoped for that firm.

What happens to a firm's data when they leave?

Deprovisioning deactivates the tenant, revokes live bank/brokerage connections, and re-homes users — while retaining all records. Verified GDPR erasure requests are the controlled exception.

Where does the data physically live?

Google Cloud, United States (us-central1), with the database on a private network and no public IP.

Are the AI's numbers real or generated text?

Real: dollar figures come from tool calls into the planning engine (Monte Carlo runs, net-worth math), carry provenance tags, and the full conversation is retained and auditable.

Are you SOC 2 certified?

Not yet, and we don't claim it. Certifications, pen-test results, and questionnaires are pilot-diligence items — ask us and we'll walk you through the current posture directly.
