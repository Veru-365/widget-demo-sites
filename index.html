For your production team

Onboard a client in four API calls

These are real endpoints from our live API, not a roadmap. During a pilot we provision partner credentials and run this alongside your team; the same flow also exists as point-and-click in our admin console if you'd rather not touch an API at all.

1Create the client's organization

One call provisions an isolated tenant, optionally on a trial clock.

curl -X POST "https://api.veru365.com/api/platform-admin/orgs?name=Acme%20Wealth&owner_user_id=1042&trial_days=30" \
  -H "Authorization: Bearer $PARTNER_TOKEN"

2Apply the client's branding

Logo and favicon upload as files; colors and product name as JSON. Instantly live across the portal, the widget, and the login page.

curl -X PATCH "https://api.veru365.com/api/orgs/47/branding" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -d '{"app_name": "Acme Wealth", "primary_color": "#0E7C5A", "secondary_color": "#0A3D2E"}'

curl -X POST "https://api.veru365.com/api/orgs/47/branding/logo" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" -F "file=@acme-logo.png"

3Choose what this client gets

Feature switches per org — bank connections, document vault, portfolio tools, scheduling — so you can package tiers however you sell them.

curl -X PATCH "https://api.veru365.com/api/platform-admin/orgs/47/features" \
  -H "Authorization: Bearer $PARTNER_TOKEN" \
  -d '{"enable_plaid": true, "enable_file_vault": true, "enable_scheduling": true}'

4Invite the advisor, drop in the widget

The invitation returns a ready-to-send link; the widget is one line on any page they own.

curl -X POST "https://api.veru365.com/api/orgs/47/invitations" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -d '{"email": "advisor@acmewealth.com", "role": "owner", "first_name": "Jordan"}'

<!-- then, on every client web page: -->
<script src="https://app.veru365.com/widget.js" data-org="acme-wealth" defer></script>

That last line shows the production form of the tag, enabled during pilot provisioning. The live demo on this site points the same tag at our staging environment — view source on any page here to see the exact running snippet.

+Events out, leads in

Outbound webhooks push platform events to your systems (signed with HMAC-SHA256 when you set a signing secret on the endpoint), and every client gets a public, rate-limited lead-capture endpoint plus a bookable public advisor page. Today advisors pull captured leads via the authenticated lead API and portal; a dedicated lead webhook event is on the roadmap.

curl -X POST "https://api.veru365.com/api/orgs/47/webhooks/endpoints" \
  -H "Authorization: Bearer $ORG_OWNER_TOKEN" \
  -d '{"url": "https://hooks.yourplatform.com/veru365",
       "secret": "whsec_yoursecret", "events": ["fact_finder.completed", "pago.*"]}'

The full REST API ships with a complete OpenAPI 3 schema — your team can generate typed clients in any language.

Being transparent: what's pilot-stage vs. shipped

  • Shipped: everything demonstrated and documented above — multi-tenant orgs, branding API, feature switches, invitations, webhooks, lead capture, the embeddable widget.
  • Provisioned with you during pilot: partner API credentials (today we issue them per-partner rather than self-serve).
